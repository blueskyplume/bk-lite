# 告警中心-相关性规则新增缺失检查 Solution Plan

版本：v1.0

需求文档：[20260312.告警中心-相关性规则新增缺失检查.md](../../requirements/告警中心/20260312.告警中心-相关性规则新增缺失检查.md)

---

## 1. 方案概览

在告警中心相关性规则中新增"缺失检查（Heartbeat Check）"策略类型，实现对心跳事件的主动监测。核心逻辑：

- 用户配置监听目标（期望正常事件的筛选条件）+ 检测周期 + 宽限期
- 系统定时检查：超时未收到预期事件时自动生成合成告警
- 事件恢复后可自动关闭告警

技术方案要点：
- 复用现有 `AlarmStrategy` 模型，扩展 `params` 字段存储缺失检查参数
- 新增 `last_execute_time` 字段作为所有策略的通用执行时间记录
- 复用 `Alert` 表存储合成告警
- 复用现有 `event_aggregation_alert` 定时任务处理缺失检查策略（与智能降噪保持一致）
- 前端表单按策略类型条件渲染不同配置步骤

---

## 2. 范围定义

### 2.1 In Scope

- 后端：`AlarmStrategy` 模型扩展（`last_execute_time` + `params` 结构）、缺失检查处理器、序列化器校验
- 后端：复用现有 `event_aggregation_alert` 定时任务集成缺失检查逻辑
- 前端：相关性规则配置表单改造，支持缺失检查类型的完整配置流程
- 数据库：新增字段迁移（`last_execute_time`）

### 2.2 Out of Scope

- 内建重复提醒策略（由分派/通知规则控制）
- 高级时序编排（连续 N 次缺失判定、按节假日排班窗口）
- 跨系统通知渠道策略改造

**核心流程**（复用现有定时任务）：

```
event_aggregation_alert (Celery Task, 每分钟)
    ├── AggregationProcessor.process_aggregation()   # 智能降噪策略
    └── HeartbeatCheckProcessor.process()            # 缺失检查策略（新增）
            ├── _get_active_strategies()  # strategy_type=missing_detection
            └── 遍历策略
                └── _process_strategy(strategy)
                        ├── _check_activation_status()  # 检查 params.heartbeat_status
                        ├── _get_matched_events()       # 按 match_rules 查询事件
                        ├── _calculate_next_expected_time()  # 计算期望时间
                        ├── _check_timeout()            # 判断是否超时
                        ├── _update_strategy_params()   # 更新 params 中的状态
                        └── _handle_timeout_or_recovery()
                                ├── 超时 → SyntheticAlertBuilder.create_alert()
                                └── 恢复 → _auto_recover_alert()
```

**状态流转口径**（存储于 `params.heartbeat_status`）：

```
waiting（待激活）
    ↓ 收到首条匹配心跳 或 立即激活
monitoring（监控中）
    ↓ 超时未收到心跳
alerting（缺失告警中）
    ↓ 收到匹配心跳（auto_recovery=True）
monitoring（已恢复，重新监控）
```

**校验规则**：

- `strategy_type = missing_detection` 时：
  - `match_rules` 必须非空（禁止 ALL 监听）
  - `params.grace_period` 必填且 > 0
  - `params.check_mode = interval` 时，`interval_value` 和 `interval_unit` 必填
  - `params.check_mode = cron` 时，`cron_expr` 必填且格式合法
  - `params.alert_template.title`、`params.alert_template.level` 和 `params.alert_template.description` 必填

- [ ] 监听目标不可选 ALL；未配置筛选条件时不可保存

### A2. 触发验收

- [ ] 立即激活模式：超过"周期 + 宽限期"未收到匹配心跳时，系统生成 1 条缺失告警

- [ ] 在"未恢复"期间持续缺失，不新增重复缺失告警记录

- [ ] 缺失告警生成后，收到任意 1 条匹配心跳，告警自动恢复（`auto_recovery` 状态）

- [ ] 已恢复后再次超时缺失，可再次触发新的缺失告警

## 5. 关键决策

| 决策项 | 选定方案 | 说明 |
|--------|----------|------|
| 心跳状态存储 | `params.heartbeat_status` 字段 | 收敛到 params JSON 字段，减少模型变更 |
| 最后执行时间 | `AlarmStrategy.last_execute_time` | 所有策略通用字段，统一记录执行时间 |
| 合成告警存储 | 复用 `Alert` 表 | 复用现有告警处置流程（分派/通知/认领等） |
| 定时任务 | 复用现有 `event_aggregation_alert` 任务 | 与智能降噪保持一致，无需新增 Celery 任务 |
| 周期计算 | `croniter` 库解析 Cron 表达式 | Python 生态成熟方案 |
| 激活方式 | `params.activation_mode` 字段 | `immediate` / `first_heartbeat` 二选一 |
| 幂等策略 | 按 `strategy_id + params.heartbeat_status=alerting` 判断 | 未恢复期间不重复生成 |

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| Cron 表达式配置错误导致大量误报 | 中 | 前端增加 Cron 校验与"下次执行时间"预览 |
| 定时任务执行时 params 更新并发冲突 | 低 | 使用数据库事务 + `select_for_update` 保证幂等 |
| 心跳事件延迟超过宽限期 | 中 | 宽限期设置提供默认值（如 5 分钟），UI 给出说明 |
| 大量规则导致定时任务执行超时 | 低 | 分批处理 + 监控任务执行时长 |
