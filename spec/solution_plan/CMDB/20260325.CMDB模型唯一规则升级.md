# CMDB 模型唯一规则升级

日期：2026-03-26

## 方案概览

本方案严格基于 PRD 与 Requirement，目标是在保留 inst_name 内置唯一能力和历史 is_only 后端兼容能力的前提下，为 CMDB 增加模型级额外唯一规则，并打通模型配置、实例写入、模型复制、配置导入导出、前端展示全链路。

总体策略：

- 保持轻量：不新增独立服务，不引入复杂规则引擎。
- 保持兼容：inst_name 继续作为内置唯一标识，历史 is_only 后端继续保留。
- 统一入口：新增规则配置入口统一收敛到模型详情的“唯一规则”页签。
- 统一校验：实例新增、修改、复制、导入、批量更新复用同一套唯一规则校验逻辑。

## 范围与约束

### In Scope

- 模型详情页新增“唯一规则”页签。
- 模型级额外唯一规则的新增、编辑、删除、查看。
- 模型级额外唯一规则的后端存储、查询与合法性校验。
- 实例新增、修改、复制、导入、批量更新统一接入额外唯一规则校验。
- 模型字段列表中唯一性展示升级。
- 字段配置页面中 is_only 收敛为只读展示，不再允许新增和编辑。
- 模型复制时复制额外唯一规则。
- 模型配置导入导出支持额外唯一规则。

### Out of Scope

- 用额外唯一规则替代或移除 inst_name 内置唯一能力。
- 恢复前端对 is_only 的新增或编辑能力。
- 支持条件唯一、跨模型唯一、按组织范围唯一。
- 支持超过 3 条额外唯一规则。
- 允许枚举、标签、布尔字段参与额外唯一规则。

## 已确认决策

- inst_name 继续保留为每个模型实例的内置业务身份字段，且唯一性不可取消、不可替换。
- 模型级额外唯一规则采用轻量存储方案，作为模型配置的一部分统一管理，不新增独立服务。
- 每个模型最多 3 条额外唯一规则。
- 每条规则支持 1 个或多个字段。
- 同一字段不能同时出现在多条额外唯一规则中。
- 额外唯一规则不支持包含 inst_name。
- 仅当前模型下的必填字段可加入额外唯一规则。
- 枚举、标签、布尔类型字段不可加入额外唯一规则。
- 新增、复制或导入唯一规则时，需要校验历史实例；若历史数据已冲突，则阻止保存。
- 规则提示采用“规则序号 + 字段组合”的轻量识别方式，不增加超出需求的复杂规则元数据。

## 数据与架构落点

### 1）模型配置层

- 在现有模型配置基础上新增 unique_rules 配置，与 attrs 同级管理。
- unique_rules 采用数组结构保存，每条规则至少包含：规则标识、字段列表、排序序号。
- 不引入独立规则表、不引入复杂关联关系，降低迁移与部署成本。

建议结构示意：

```json
[
  {
    "rule_id": "rule_1",
    "field_ids": ["ip", "port"],
    "order": 1
  },
  {
    "rule_id": "rule_2",
    "field_ids": ["serial_no"],
    "order": 2
  }
]
```

### 2）实例校验层

- 保留现有 inst_name 与历史 is_only 的单字段唯一校验。
- 在现有校验基础上增加模型级联合唯一规则校验。
- 所有实例写入场景统一装载模型唯一规则，避免不同入口口径不一致。

### 3）前端交互层

- 在模型详情页新增独立“唯一规则”页签，承载全部模型级唯一规则配置操作。
- 字段配置页面中 is_only 不再作为可编辑能力暴露，仅保留只读展示。
- 字段列表中的唯一性展示分为：非唯一、单字段唯一、联合唯一。

## 分阶段计划

### 阶段一：模型级唯一规则配置能力落地

**里程碑目标**

- 完成模型级额外唯一规则的存储、接口与保存校验。

**交付内容**

- 模型配置中新增 unique_rules 数据结构。
- 新增模型级唯一规则的查询、新增、编辑、删除接口。
- 保存规则时执行完整合法性校验：
  - 最多 3 条规则。
  - 规则中字段不可重复出现在其他规则。
  - 不可包含 inst_name。
  - 字段必须属于当前模型。
  - 字段必须为必填。
  - 字段类型不能为 enum、tag、bool。
- 新增规则保存前扫描历史实例，若存在冲突数据则阻止保存。
- 模型字段删除、修改必填属性、修改字段类型等操作前增加依赖检查，若会破坏规则合法性则阻止保存。

**里程碑结果**

- 模型可维护额外唯一规则。
- 模型字段变更不会破坏已配置规则的一致性。

### 阶段二：实例写入统一校验落地

**里程碑目标**

- 实例所有写入入口统一执行 inst_name 内置唯一校验与模型级额外唯一规则校验。

**交付内容**

- 在实例新增、修改、复制链路中统一加载 unique_rules。
- 在批量更新链路中增加模型级唯一规则校验。
- 在实例导入链路中增加模型级唯一规则校验。
- 增加批次内互斥检查，避免导入或批量更新时同批数据彼此冲突。
- 错误提示统一输出“规则序号 + 字段组合 + 冲突信息”。

**里程碑结果**

- 新增、修改、复制、导入、批量更新场景行为一致。
- 任何违反规则的数据均无法写入。

### 阶段三：前端配置与展示收敛

**里程碑目标**

- 完成模型管理侧唯一规则配置页面、字段唯一性只读展示与 is_only 编辑入口收敛。

**交付内容**

- 模型详情新增“唯一规则”页签。
- 页签支持规则列表、创建、编辑、删除。
- 字段候选范围仅限当前模型可参与唯一规则的字段。
- 已被其他规则占用的字段在当前配置中不可再次选择。
- 已满 3 条规则时，“新增规则”按钮禁用。
- 字段配置弹窗中移除或禁用 is_only 编辑能力，仅保留只读信息展示。
- 字段列表“唯一性”列升级：
  - 历史 is_only 展示为单字段唯一。
  - 被单字段额外唯一规则使用的字段展示为单字段唯一。
  - 被多字段规则使用的字段展示为联合唯一。

**里程碑结果**

- 用户可在统一入口中维护规则。
- 前端不会再出现字段配置页与模型规则页双入口并存的混乱。

### 阶段四：模型复制、配置迁移与上线准备

**里程碑目标**

- 打通模型复制、模型配置导入导出与最终发布路径。

**交付内容**

- 模型复制时，如复制模型属性，则同步复制 unique_rules。
- 复制后的规则按目标模型字段重新执行合法性校验。
- 模型配置导出包含 unique_rules。
- 模型配置导入支持 unique_rules，并执行字段合法性与历史实例冲突校验。
- 完成前后端联调、回归测试、上线与回滚预案确认。

**里程碑结果**

- 规则能力覆盖模型复制与配置迁移场景。
- 具备安全上线条件。

## 阶段验收清单

- 模型详情页存在独立“唯一规则”页签。
- 用户可在该页签查看、新增、编辑、删除额外唯一规则。
- 每个模型最多只能配置 3 条额外唯一规则。
- 每条额外唯一规则支持由 1 个或多个字段组成。
- 额外唯一规则中不可选择 inst_name。
- 同一字段不可同时出现在多条额外唯一规则中。
- 仅必填字段可以被加入额外唯一规则。
- 枚举、标签、布尔类型字段不可加入额外唯一规则。
- 字段删除、字段取消必填、字段类型调整等会破坏规则合法性的操作会被阻止。
- 历史 is_only 前端仅只读展示，不再允许编辑。
- 字段列表中，参与多字段唯一规则的字段显示为“联合唯一”。
- 实例新增、修改、复制、导入、批量更新均会同时校验 inst_name 与额外唯一规则。
- 任一唯一规则冲突时，系统阻止写入，并返回可识别的冲突提示。
- 复制模型时，复制属性场景会同步复制额外唯一规则。
- 模型配置导入导出支持额外唯一规则配置。
- 新增、复制或导入规则时，如历史实例冲突，则阻止保存。

## 风险与应对

- 风险：实例写入入口多，规则实现分叉后容易出现脏数据。
  应对：统一在实例服务层与导入工具层复用同一套规则装载与校验逻辑。

- 风险：新增规则时历史实例数据量较大，扫描成本上升。
  应对：本期先基于现有查询能力做保存前扫描；规则上限仅 3 条，优先保证实现简单可靠。

- 风险：前端仍保留 is_only 编辑入口，造成双入口配置混乱。
  应对：本期明确收敛前端字段配置中的 is_only 编辑能力，唯一规则统一从模型详情页签维护。

- 风险：模型配置导入导出格式扩展后影响旧文件兼容。
  应对：导入时兼容缺失 unique_rules 的旧文件，按空规则处理。

- 风险：批量导入或批量更新时，只校验库内冲突而漏掉批次内冲突。
  应对：统一增加批次内互斥检查。

## 依赖与协同

- 产品：确认页签命名、字段列表唯一性展示文案、冲突提示文案。
- 后端：负责模型配置扩展、规则合法性校验、实例统一校验、复制与导入导出改造。
- 前端：负责唯一规则页签、字段候选过滤、字段唯一性展示升级、is_only 编辑入口收敛。
- 测试：覆盖模型配置、字段改造阻断、实例写入、导入导出、模型复制全链路回归。
